Visión General

¿Qué es Ticket Monster?

Sistema en línea de venta de tickets para eventos de gran escala. Diseñado para soportar aperturas de venta masivas con picos de decenas de miles de usuarios simultáneos.

Monolito modular event-driven con Spring Modulith. Cada módulo mapea a un bounded context de DDD con comunicación híbrida (síncrona + asíncrona).

Garantía de cero overbooking mediante locks distribuidos Redis + PostgreSQL SELECT FOR UPDATE.

Pilares Técnicos

  • ● Alta Concurrencia — Fila virtual Redis + rate limiting en Traefik
  • ● Consistencia CP — Anti-overbooking con doble lock
  • ● Observabilidad — Grafana + Prometheus + Loki + Tempo
  • ● Event-Driven — Redpanda para comunicación asíncrona
  • ● Cloud Native — K3s + Helm + Docker

07
Mecanismo Crítico

Anti-Overbooking

1. Validación de Límite

Máximo 3 tickets por usuario por evento. Previene acaparamiento.

2. SELECT FOR UPDATE (PostgreSQL)

Bloquea la fila de zone_stock. Solo 1 transacción toca la fila a la vez. Serializa todas las peticiones concurrentes sobre la misma zona.

3. Redis Lock (SET NX EX 600)

Previene que el mismo usuario haga doble reserva en la misma zona. Key: reservation:{event}:{zone}:{user}.

4. Keyspace Notifications

Redis notifica expiración del lock (TTL 10 min). El sistema repone el stock automáticamente: UPDATE zone_stock SET available = available + N.

Resultado: Cero Overbooking

Test de contención: 1000 VUs compitiendo por 10 tickets. Resultado: exactamente 10 reservas creadas, 990 rechazos 409. 0% overbooking.

100% Success rate
08
Fila Virtual

Redis FIFO Queue

Mecanismo

Usuarios entran con LPUSH a una cola Redis FIFO. Un dispatcher procesa en batch: 500 usuarios cada 2 segundos con RPOP.

Usuarios marcados como TURN_READY reciben un JWT de 5 min TTL para acceder al módulo de reservas.

Expiración sin Uso

Si el JWT expira antes de que el usuario reserve, el slot se libera automáticamente y se reasigna al siguiente en la cola.

User 2 LPUSH
User N LPUSH

Decisión de Arquitectura

¿Por qué se eliminó el API Gateway?

Antes: Spring Cloud Gateway

El API Gateway introducía un hop de red adicional entre Traefik y el monolito. Bajo carga alta, el connection pooling del gateway se saturaba, añadiendo latencia significativa.

Hallazgo k6

Load tests revelaron que el overhead de conexión del Gateway era el cuello de botella principal en escenarios de alta concurrencia (>1000 VUs).

Ahora: Traefik Directo

Traefik Ingress (nativo de K3s) proporciona TLS termination, rate limiting y resilience en el edge. El monolito recibe tráfico directo sin hop extra.

"Eliminar el extra network hop redujo connection churn y latencia bajo carga extrema."

Edge

Rate Limiting Strategy

ticketmonster-graphql

Catch-all para /graphql. Sin rate limiting. Las queries de lectura son públicas y baratas (MongoDB).

ticketmonster

/api/* y /actuator/*. Con rate limiting + secure headers. Solo aplica a endpoints de escritura (reservations, payments, queue joins).

Ventajas

  • ● Catálogo siempre disponible sin throttling
  • ● Protección en escrituras críticas (anti-abuso)
  • ● Dos Ingress resources independientes en Traefik

Mecanismo de Lock

Redis lock (SETNX): 100 usuarios distintos → cada uno pasa su lock sin conflicto (keys distintas).

PostgreSQL SELECT FOR UPDATE: 1000 transacciones serializadas sobre la misma fila. Solo 10 ven availableCount ≥ 1.

0 overbooking con 10 tickets y 1000 peticiones. El doble lock cumple su función.
