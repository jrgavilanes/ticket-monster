Sección 1

Introducción y contexto de negocio

Sistema en línea de venta de tickets para eventos de gran escala, diseñado para soportar picos extremos de demanda durante aperturas de venta masivas.

Gestiona múltiples venues, tipos de evento (conciertos, deportes, teatro, festivales, conferencias), fechas y artistas. Los usuarios consultan el catálogo, se unen a una fila virtual, reservan entradas por zona con TTL de 10 minutos y confirman el pago.

Flujo intencionadamente separado en 6 etapas explícitas — cada una escala y falla de forma independiente. La fila virtual absorbe la avalancha inicial sin tocar la base de datos de reservas.

50MDAUUsuarios diarios activos
5MCCUConcurrentes en apertura
0 Overbooking Bajo cualquier condición de concurrencia

FLUJO DE COMPRA

1. Unirse a cola virtual → 2. Esperar turno → 3. Obtener token → 4. Reservar entradas → 5. Iniciar pago → 6. Confirmar pago

Sección 3

Visión general — Modelo C4

Monolito modular event-driven con Spring Modulith. Cada módulo mapea a un bounded context DDD.

Nivel 1 — Contexto

Usuario → Traefik (TLS + Rate Limit) + Keycloak (OAuth2) → Ticket Monster. Sistema autocontenido sin dependencias cloud externas.

Nivel 2 — Contenedores

4 módulos en el monolito: Catalog (GraphQL + MongoDB), Virtual Queue (REST + Redis), Reservation (REST + PostgreSQL + Redis), Payment (REST + PostgreSQL). Mensajería vía Redpanda. Observabilidad con Loki + Prometheus + Tempo + Grafana.

Nivel 3 — Componentes

Cada módulo expone sus controladores, servicios, repositorios y publishers. Comunicación híbrida: síncrona para queries (interfaces Java) y asíncrona para eventos de dominio (Redpanda).

Bases de datos aisladas por bounded context: schemas PostgreSQL separados (reservation, payment) y prefijos Redis diferenciados (queue:, reservation:) — preparan extracción futura a microservicios sin migración de datos.

Sección 4

Descomposición DDD + Context Map

Catalog AP · Supporting

Catálogo de eventos, venues y artistas. MongoDB (documental, schema flexible). GraphQL público. Única dependencia síncrona: consulta availability a Reservation.

Virtual Queue AP · Core

Cola FIFO en Redis. Absorbe avalancha sin tocar PostgreSQL. Batch dispatcher limita entrada a 250 usuarios/s. Token HMAC como llave de acceso sin dependencia de Keycloak.

Reservation CP · Core

Núcleo transaccional. Doble lock (Redis SETNX + PostgreSQL SELECT FOR UPDATE). Guardián del cero overbooking. Publica eventos en Redpanda.

Payment CP · Supporting

Simulación con idempotencia y auditoría completa. payment-confirmed en Redpanda convierte reserva a SOLD.

Sección 7

Mecanismo anti-overbooking

Tres capas de defensa superpuestas que garantizan cero overbooking incluso con 1000+ peticiones simultáneas compitiendo por las mismas entradas.

Capa 1 — Límite de tickets por usuario

Verifica que el usuario no exceda el máximo configurable (3 tickets por evento). Protección anti-abuso que reduce contención en capas siguientes.

SELECT COUNT(*) WHERE user_id = ? AND status = 'ACTIVE'
Capa 2 — SELECT FOR UPDATE (PostgreSQL)

Bloqueo pesimista a nivel de fila serializa todas las peticiones concurrentes. 1000 peticiones → ejecución secuencial sobre zone_stock. La undécima encuentra available_count = 0 → ROLLBACK → 409.

BEGIN → SELECT FOR UPDATE → UPDATE available_count → INSERT → COMMIT
Capa 3 — Lock distribuido (Redis SETNX)

Cierra la brecha del mismo usuario con peticiones concurrentes. Key reservation:{eventId}:{zoneId}:{userId} con TTL 10 min. Solo una petición adquiere el lock.

SETNX reservation:{eId}:{zId}:{uId} EX 600

Verificación empírica (k6): 1000 peticiones concurrentes, 100 usuarios, 10 tickets en zone-vip → exactamente 10 reservas creadas (201), 990 rechazos (409), 0 errores, 0 overbooking.

Estrategia de Fila Virtual

Respuesta arquitectónica al problema de 5M de usuarios concurrentes en apertura de venta.

1

LPUSH

Usuario se encola en Redis FIFO. O(1), microsegundos. Idempotente vía SET auxiliar.

POST /join
2

Polling

Cliente consulta posición periódicamente. LRANGE + HEXISTS.

GET /status
3

Batch Dispatcher

@Scheduled cada 2s extrae 500 usuarios (RPOP). Limita tasa a 250/s.

HSET TURN_READY
4

Token HMAC

Firma HMAC-SHA256. TTL 5 min. Sin dependencia de Keycloak.

GET /token
5

Reserva

Usuario presenta token al Reservation Module por HTTP.

POST /reservations
AspectoRedisKafka/Redpanda
Latencia< 1ms~5-10ms
Throughput~100K ops/s~1M msg/s (+ overhead)
Modelo datosLIST nativa FIFOLog inmutable
Posición usuarioLLEN O(1)Requiere contador externo
ComplejidadMínimaConsumidores, offsets

Trade-off aceptado: si Redis se cae, la cola se vacía. Usuarios reintentan. Es AP deliberado — se prioriza rendimiento sobre durabilidad. Para eventos de dominio críticos se usa Redpanda (CP).

Resultado k6: 526 req/s, p95 676ms, 0% errores con 200 VUs concurrentes sobre K3s remoto (1 pod).

Comunicación asíncrona — Redpanda

Redpanda (Kafka-compatible, Raft consensus) elegido sobre Apache Kafka por menor huella operacional y latencias más bajas en despliegues pequeños.

TópicoProductorConsumidor
reservation-createdReservationPayment (+ Notificaciones futuro)
reservation-cancelledReservationPayment (reembolso futuro)
reservation-expiredReservationPayment (reembolso futuro)
payment-confirmedPaymentReservation (→ SOLD)
payment-refund-requiredReservationPayment (futuro)

Garantías de entrega

  • acks=all — confirmación de todas las réplicas
  • enable.idempotence=true — exactly-once por partición
  • Consumidores con commit manual — at-least-once

FLUJO DE EVENTOS — COMPRA EXITOSA

1. Reservation → reservation-created → Redpanda
2. Redpanda → reservation-created → Payment
3. Payment → payment-confirmed → Redpanda
4. Redpanda → payment-confirmed → Reservation
↳ Convierte reserva a SOLD

Justificación: desacoplamiento temporal (Reservation no espera a Payment), resiliencia (eventos se acumulan si el consumidor está caído), y preparación para extracción a microservicios sin cambiar el protocolo de comunicación.

Evolución síncrona → gRPC: la única llamada síncrona actual (Catalog → Reservation para availability) se migrará a gRPC con contrato .proto al extraer Catalog. Eventos internos Spring → tópico Redpanda zones-modified.

Sección 10

Seguridad — Defensa en 4 capas

Capa 1 — Edge (Traefik)

  • TLS termination (Let's Encrypt)
  • Rate limiting: 10K req/s en /api/*
  • Secure headers (HSTS, XSS, etc.)
  • 2 Ingress separados: GraphQL público + API con rate limit

Decisión: eliminar API Gateway Spring Cloud → Traefik asume sus responsabilidades. Pruebas k6 mostraron que el gateway añadía ~50-100ms y era punto único de fallo.

Capa 2 — Auth (Keycloak)

  • Realm ticket-monster
  • OAuth2/OIDC con JWT RS256
  • Roles: USER, ADMIN
  • Clientes: app + Grafana

Monolito actúa como OAuth2 Resource Server validando JWT contra Keycloak.

Capa 3 — Autorización (Spring)

  • GraphQL queries: públicas
  • GraphQL mutations: ADMIN
  • REST /api/v1/*: JWT válido
  • Autoría de recursos: filtro por userId del JWT

Capa 4 — Negocio

  • Token HMAC de cola (sin Keycloak)
  • Idempotencia de pagos
  • Límite 3 tickets/usuario
  • Locks Redis por usuario+zona

Sección 11

Resiliencia y concurrencia

Circuit Breakers — Resilience4j

3 breakers independientes: catalog, reservation, payment. Ventana deslizante 10 llamadas, umbral 50%, estado abierto 10s. Time limiter 5s. Rate limiter global: 100 req/min.

Virtual Threads (Java 21)

Millones de threads virtuales con código imperativo. I/O-bound: el thread se desmonta durante bloqueo y se reanuda al recibir respuesta. Stack traces lineales y legibles.

Fallback multinivel

  • Circuit breaker abierto → HTTP 503 controlado (no timeout)
  • Redis caído → SETNX falla, reservas rechazadas (409)
  • Keyspace notifications perdidas → barrido periódico cada 60s
  • Redpanda caído → eventos se acumulan, se procesan al recuperar

Sección 12

Análisis CAP Theorem

El sistema no adopta una única postura CAP. Cada bounded context elige la estrategia adecuada a su caso de uso.

Reservation
CP

Cero overbooking no negociable. SELECT FOR UPDATE + SETNX garantizan consistencia.

Payment
CP

Transacciones financieras requieren consistencia absoluta. Idempotencia vía UNIQUE.

Catalog
AP

Read-heavy. Inconsistencia temporal (ms) aceptable. Disponibilidad de lectura priorizada.

Virtual Queue
AP

Redis standalone. Cola se pierde si cae — usuarios reintentan. CP no soportaría 500K+ ops/s.

Redpanda
CP

Raft consensus. Eventos de dominio no se pierden ni duplican durante partición.

Caída PostgreSQL: Reservation deja de aceptar (503). Circuit breakers abren. Usuarios esperan en cola (Redis AP). Al recuperar, el sistema reanuda.

Caída Redis: Cola desaparece (AP). Locks fallan → nuevas reservas rechazadas (409). Reservas existentes no se corrompen.

Partición de red: Catalog→Reservation: fail-close. Reservation↔Payment: eventos acumulados en Redpanda, Raft garantiza no pérdida.

Estrategia de evolución — Monolith First

Principio de Monolith First (Martin Fowler): comenzar como monolito modular y extraer microservicios solo cuando las necesidades del negocio lo justifiquen.

Criterio de extracciónEjemplo en Ticket Monster
Escalado independienteCatalog: 10 réplicas lectura vs Reservation: 2 réplicas + CPU
Ciclos de releaseCatalog: cambios diarios vs Payment: despliegues mensuales
Tecnología específicaPayment → Node.js (Stripe). Queue → Go (mínima latencia)
Aislamiento de fallosMemory leak en Catalog no debe tumbar reservas activas
Escalado organizacionalMúltiples equipos con propiedad exclusiva

Preparación actual (3 elementos clave)

  • Comunicación asíncrona vía Redpanda ya establecida
  • Schemas PostgreSQL separados por bounded context
  • Interfaces para dependencias síncronas (AvailabilityService → gRPC futuro)

ROADMAP TENTATIVO

Fase 1 — Actual

Monolito modular con Spring Modulith. Verificación arquitectónica automática. Validar producto y flujos de negocio.

Fase 2 — Extracción Catalog

Trigger: throughput lectura > 10K req/s. Microservicio Spring Boot + MongoDB dedicado. gRPC para availability, Redpanda para eventos.

Fase 3 — Extracción Payment

Trigger: integración pasarela de pago real. Microservicio independiente (posiblemente otro lenguaje). Comunicación vía Redpanda sin cambios.

Fase 4 — Extracción Virtual Queue

Trigger: latencia < 10ms p99 para 1M+ usuarios. Servicio en Go/Rust pegado a Redis. Protocolo token HMAC se mantiene.

Decisiones arquitectónicas y trade-offs

API Gateway → Traefik

Gateway añadía ~50-100ms y era punto único de fallo. Se delegó TLS, rate limiting, CORS a Traefik. Trade-off: pérdida de granularidad por endpoint.

Token HMAC vs JWT Keycloak

HMAC local tarda µs vs decenas de ms de HTTP a Keycloak. Sin dependencia externa en path caliente. Trade-off: no es OAuth2 estándar.

Redis SETNX vs PostgreSQL advisory locks

TTL automático sin locks huérfanos. Keyspace notifications restauran stock. Trade-off: Redis es dependencia externa adicional en path crítico.

MongoDB + PostgreSQL (políglota)

Schema flexible para catálogo + ACID para transacciones. Trade-off: dos motores → doble backup, doble configuración operacional.

Virtual Threads vs WebFlux

Código imperativo legible con mismo rendimiento I/O no bloqueante. Stack traces lineales. Trade-off: sin ventaja en CPU-bound puro.

Rate limiting: edge vs app

Traefik rechaza antes de llegar al monolito. Resilience4j como backstop intra-app. Trade-off: granularidad por Ingress, no por endpoint individual.

Filosofía: cada decisión documenta evidencia (pruebas k6), justificación técnica y contrapartida aceptada. No hay decisiones perfectas — solo trade-offs explícitos.
