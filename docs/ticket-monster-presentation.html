Cuestionario de evaluación

Anexo A · 5 cuestiones
A.1 · ANTI-OVERBOOKING ¿Cómo previene el sistema el overbooking?
3 capas de defensa. Capa 1: límite de 3 tickets por usuario. Capa 2: SELECT FOR UPDATE en zone_stock serializa todas las transacciones sobre la misma zona. Capa 3: SETNX en Redis (`reservation:{event}:{zone}:{user}`) cierra la brecha del mismo usuario en peticiones concurrentes. Expiración automática vía keyspace notifications + sweeper cada 60 s. Validado con k6: 1.000 VUs por 10 tickets → exactamente 10 reservas, 990 rechazos, 0 overbooking.
A.2 · RESILIENCIA ¿Dónde y cómo se integra la resiliencia?
4 niveles coordinados. Edge (Traefik): rate limit 10K req/s + TLS + secure headers. App (Resilience4j): 3 circuit breakers independientes (catalog/reservation/payment), time limiter 5 s. Datos: locks Redis con TTL de 10 min auto-expirables. Mensajería: Redpanda con acks=all + enable.idempotence=true + commit manual. Fallback: ReservationExpirationSweeper si fallan las notificaciones.
A.3 · CONCURRENCIA ¿Cómo se usa la concurrencia?
4 dimensiones. Reservas: PostgreSQL serializa con FOR UPDATE → ~250 reservas/s por zona, suficiente porque la fila virtual limita la entrada. Cola: Redis single-threaded serializa sin locks explícitos. Monolito: Java 21 Virtual Threads permiten millones de threads concurrentes con código imperativo. Pagos: idempotency_key UNIQUE garantiza un solo efecto por confirmación concurrente.
A.4 · LLEGADA MASIVA ¿Cómo se manejan 5M usuarios concurrentes?
Tres mecanismos coordinados. 1) Fila virtual Redis (LPUSH O(1), microsegundos) absorbe la avalancha sin tocar PostgreSQL. 2) Batch dispatcher extrae 500 usuarios cada 2 s → limita entrada a 250 usuarios/s. 3) Rate limiting en Traefik (HTTP 429) antes de que la petición consuma recursos. Flujo en pico: 5M usuarios → Traefik → Redis (cientos de miles LPUSH/s) → Dispatcher (250/s) → SELECT FOR UPDATE → 201 o 409.
A.5 · TEOREMA CAP ¿Cómo se aplica el CAP Theorem?
Estrategia híbrida por bounded context. Reservation y Payment son CP (consistencia crítica: cero overbooking, integridad financiera). Catalog y Virtual Queue son AP (read-heavy, toleran eventual consistency para ganar rendimiento). Redpanda es CP (Raft consensus para eventos de dominio). Imponer CP global sacrificaría rendimiento; imponer AP global provocaría overbooking. Diferentes partes del sistema tienen diferentes requisitos de consistencia.
