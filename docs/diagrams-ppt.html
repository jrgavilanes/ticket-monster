1.1

Alcance del Sistema

Ticket Monster es un sistema en linea de venta de tickets para eventos de gran escala, disenado para soportar picos extremos de demanda durante aperturas de venta masivas, garantizando cero overbooking.

Gestion de multiples venues, tipos de evento, fechas y artistas. Flujo: catalogo → fila virtual → reserva (TTL 10 min) → pago confirmado. Disenado como monolito modular event-driven con Spring Modulith, preparado para extraccion a microservicios.

Modulos del Monolito

Catalog Module
GraphQL · MongoDB · Solo Lectura
API GraphQL publica para consultar eventos, venues, artistas y disponibilidad. Unico modulo con acceso a MongoDB.
Virtual Queue Module
REST · Redis · FIFO
Cola FIFO sobre Redis. Absorbe la avalancha de usuarios en aperturas sin tocar PostgreSQL. Alta disponibilidad (AP).
Reservation Module
REST · PostgreSQL + Redis · Anti-Overbooking
Nucleo transaccional CP. Doble lock (Redis SETNX + PostgreSQL SELECT FOR UPDATE). Publica eventos en Redpanda.
Payment Module
REST · PostgreSQL · Idempotencia
Gestion del ciclo de vida del pago con idempotencia y auditoria completa. Publica payment-confirmed en Redpanda.

Anti-Overbooking

La invariante de negocio mas critica del sistema

7

Mecanismo Anti-Overbooking — 3 Capas

sequenceDiagram actor U as Usuario participant R as ReservationService participant PG as PostgreSQL participant Redis as Redis U->>R: POST /reservations (N tickets, zona Z) Note over R: Capa 1: Limite por usuario R->>PG: SELECT COUNT(*) WHERE status = ACTIVE PG-->>R: count Note over R,PG: Capa 2: Bloqueo pesimista R->>PG: BEGIN TRANSACTION R->>PG: SELECT FOR UPDATE zone_stock PG-->>R: available_count alt available_count >= N Note over R,Redis: Capa 3: Lock Redis SETNX R->>Redis: SETNX lock EX 600 Redis-->>R: OK R->>PG: UPDATE stock - COMMIT R-->>U: 201 Created else Sin stock R->>PG: ROLLBACK R-->>U: 409 Conflict end Note over Redis,R: Expiracion (10 min) Redis-->>R: Keyspace EXPIRE R->>PG: Restaurar stock + EXPIRED

Tres capas de defensa: (1) Limite 3 tickets/usuario, (2) SELECT FOR UPDATE serializa acceso a zone_stock, (3) Redis SETNX evita peticiones duplicadas del mismo usuario. Barrido periodico cada 60s como fallback.

7

Como Funciona el Anti-Overbooking

01
Limite por Usuario
Max 3 tickets por evento. Proteccion anti-abuso y anti-reventa. Reduce contencion en capas siguientes.
02
SELECT FOR UPDATE
Pieza central. Bloquea la fila de zone_stock. 1000 peticiones se ejecutan secuencialmente. Undecima encuentra stock=0.
03
Redis SETNX
Cierra brecha: mismo usuario, peticiones concurrentes. Solo una adquiere el lock. TTL 10 min anti-huerfanos.
Evidencia Empirica (k6)
1000 peticiones concurrentes · 100 usuarios · 10 tickets en zone-vip → exactamente 10 reservas creadas (201), 990 rechazos (409), 0% errores, 0 overbooking.

Estrategia de Fila Virtual

graph TD U1["Usuario 1"] -->|"LPUSH"| Q["Redis FIFO Queue - queue:{eventId}"] U2["Usuario 2"] -->|"LPUSH"| Q UN["Usuario N"] -->|"LPUSH"| Q Q -->|"RPOP batch - 500 usuarios / 2s"| D["Dispatcher - @Scheduled"] D -->|"HSET TURN_READY"| TR["Turn Ready Hash - queue:turn:{eventId}"] TR -->|"GET /token"| JWT["Token HMAC - 5 min TTL"] JWT -->|"presentado por cliente"| RES["Reservation Module"] JWT -->|"expirado sin uso"| FREE["Liberar Slot"] FREE --> D

Respuesta a 5M de usuarios concurrentes: FIFO en Redis (LPUSH O(1), microsegundos). Batch dispatcher limita tasa de entrada a PostgreSQL a 250 usuarios/s. Token HMAC de 5 min evita dependencia de Keycloak en el path caliente.

Evolucion y Trade-offs

Roadmap Monolith First
Fase 1 — Actual
Monolito modular Spring Modulith. Comunicacion asincrona Redpanda. Verificacion arquitectonica automatica.
Fase 2 — Catalog
Trigger: throughput > 10K req/s. Extraer a microservicio con MongoDB dedicado. gRPC para availability.
Fase 3 — Payment
Trigger: integracion pasarela real. Extraer a servicio independiente. Redpanda sin cambios.
Fase 4 — Queue
Trigger: latencia < 10ms p99. Servicio en Go/Rust pegado a Redis.
Decisiones Clave
API Gateway → Traefik
Eliminado por latencia + SPOF. Traefik asume TLS, rate limiting, secure headers.
Token HMAC vs JWT Keycloak
HMAC local (microsegundos) vs HTTP a Keycloak (decenas de ms). Sin dependencia externa en path caliente.
Virtual Threads vs WebFlux
Operaciones secuenciales con codigo imperativo. Mismo rendimiento I/O sin complejidad reactiva.
MongoDB vs PostgreSQL catalogo
Schema flexible + full-text search nativo. Trade-off: dos motores de BD = doble operacion.
